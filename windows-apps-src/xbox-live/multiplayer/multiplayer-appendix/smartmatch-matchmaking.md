# <a name="smartmatch-matchmaking"></a>스마트 매치 매치 메이킹

이 문서는 다음 섹션
* 스마트 소개
* 스마트 런타임 작업
* 타이틀에 대 한 스마트 구성
* 스마트 구성 하는 동안 팀 규칙을 정의합니다.
* 대상 세션 초기화 및 QoS

## <a name="introduction-to-smartmatch"></a>스마트 소개

XSAPI 라는 [멀티 플레이어 관리자 API](../multiplayer-manager/multiplayer-manager-api-overview.md)를 래핑하는 스마트 매치 메이 킹 서비스를 제공 합니다.  고급 API 사용에 대 한 **MatchmakingService 클래스**참조할 수 있지만 멀티 플레이어 관리자를 사용 하 여 구현 하는 것이 불가능 매치 메이 킹 시나리오가 있을 경우 하십시오 피드백을 제공 해 주세요에 댐을 통해 합니다.  사용 하는 API, 관계 없이이 문서의 개념 정보 적용 됩니다.

스마트 매치 메이 킹 함께 재생 하려는 사용자 정보 및 사용자에 대 한 연결 요청에 따라 플레이어를 그룹화 합니다. 매치 메이 킹은 서버 기반 사용자가 서비스에 요청을 제공 하 고 나중에 관한 알림을 일치 하는 경우를 의미 합니다. Xbox One 용 타이틀을 빌드하는 경우에 [스마트 매치를 사용 하 여](using-smartmatch-matchmaking.md)에 설명 된 대로 스마트를 사용할 수 있습니다. 또는 자체 매치 메이 킹 서비스 [자체 매치 메이 킹 서비스를 사용 하 여](https://developer.microsoft.com/en-us/games/xbox/docs/xboxlive/xbox-live-partners/multiplayer-and-networking/using-your-own-matchmaking-service)에 설명 된 대로 로드할 수 있습니다. 이 링크에 액세스 해야 한다는 Xbox Live 개발에 사용 되는 Microsoft 개발자 센터 계정 있다고 참고 합니다.

### <a name="about-smartmatch"></a>스마트 정보

매치 메이 킹 서비스 연결을 단순화 하기 위해 MPSD 밀접 하 게 작동 합니다. 타이틀 예를 들어 사용자는 싱글 플레이어 재생 되는 동안 백그라운드에서 연결을 쉽게 할 수 있습니다 제목 내 합니다.

개인 또는 그룹 매치 메이 킹 입력 하고자 하는 일치 티켓 세션을 만든 다음 함께 작업할 일치 하는 설정 다른 플레이어를 찾으려고 매치 메이 킹 서비스 요청 합니다. 따라서 임시 매치 메이 킹 서비스 기간에 대 한 내에 있는 "일치 티켓"의 생성 됩니다.

매치 메이 킹 서비스 구성에 따라 함께 재생 세션 선택 통계 각 플레이어와 일치 요청 시 지정 된 추가 정보를 저장 합니다. 그런 다음 서비스를 일치 하 고 일치 하는 사용자의 제목을 알립니다 모든 플레이어를 포함 하는 일치 대상 세션을 만듭니다.

대상 세션 준비 되 면 타이틀 그룹 함께 재생 하 고 모든 경우에 재생을 시작할 수 있는지 확인 하려면 QoS 검사를 수행 합니다. QoS 프로세스 및 matchmade 게임 플레이 하는 동안 타이틀 세션 상태 MPSD, 내에서 최신 상태로 유지 하 고 세션에 변경에 대 한 MPSD에서 알림을 받을 합니다. 그러한 변경 사항을 들어오고, 사용자를 포함 하 고 세션 중재자로 변경 합니다.


### <a name="match-ticket-session"></a>일치 티켓 세션

일치 티켓 세션 일치 여부를 확인 하려면 플레이어에 대 한 클라이언트를 나타냅니다. 에 게임 또는 함께 로비 있는 낯선 사람 그룹에서 또는 플레이어의 다른 제목 특정 그룹에 따라 만들어집니다. 경우에 따라 티켓 세션 더 많은 플레이어 추구 하는 진행 중인 이미 게임 세션 수도 있습니다.


### <a name="match-ticket"></a>티켓 일치

매치 메이 킹 시도 추적 하는 일치 티켓 만들기 매치 메이 킹 티켓 세션을 제출 하면 됩니다. 예를 들어 게임 지도 또는 플레이어의 수준을 티켓 세션에서 플레이어의 특성과 함께 티켓 특성 일치를 확인 하는 데 사용 됩니다.
#### <a name="hoppers"></a>Hoppers

Hoppers는 일치 티켓을 수집 하는 논리적이 좋습니다. 동일한 hopper 내 티켓만 연결할 수 있습니다. 타이틀 여러 hoppers 있을 수 있습니다. 예를 들어 제목을 하나의 hopper 어떤 플레이어에 대 한 기술은 일치 하는 가장 중요 한 항목을 만들 수 있습니다. 다른 hopper 있는 플레이어만 일치 하는 동일한 다운로드 가능한 콘텐츠를 구매한 경우 사용할 수 있습니다.


#### <a name="hopper-rules"></a>Hopper 규칙

Hopper 규칙 매치 메이 킹 서비스 함께 그룹화 할 플레이어에서 결정 하는 데 사용 하는 조건을 정의 제공 합니다. 규칙의 두 가지: MUST 규칙-호환 고려해 야 할 일치 티켓에 대 한 충족 해야 합니다.
-   규칙은-일치 티켓 규칙 일치 하지 않는 한 선호 됩니다.

각이 범주 내에서 규칙의 몇 가지 특정 형식이 있습니다. 자세한 내용은 **스마트 런타임 작업**의 런타임 작업 구성 정보를 참조 하세요.


### <a name="match-target-session"></a>일치 대상 세션

일치 하는 그룹을 찾으면 서비스 일치 대상 세션을 만들고 일치 하는 함께 티켓 세션에서 모든 플레이어에 대 한 장소를 예약 합니다.

### <a name="setting-matchmaking-attributes-on-the-session-and-players"></a>세션 및 플레이어에 매치 메이 킹 특성을 설정합니다.

티켓 세션을 제출할 제목 세션을 설정 하는 플레이어 특성이 때 다른 세션을 사용 하 여 세션을 그룹화 할 매치 메이 킹 서비스 사용 합니다. 티켓 수준 또는 구성원 단위 수준에서 특성을 지정할 수 있습니다.


### <a name="setting-matchmaking-attributes-at-the-match-ticket-level"></a>매치 메이 킹 특성 일치 티켓 수준 설정

제목은 MatchmakingService.CreateMatchTicketAsync 메서드의 ticketAttributesJson 매개 변수에서 일치 티켓 수준에서 특성을 제출합니다. 티켓 수준에서 지정 된 특성 구성원 단위 수준에서 지정 된 동일한 특성을 재정의 합니다.


#### <a name="setting-matchmaking-attributes-at-the-per-member-level"></a>구성원 단위 수준에서 매치 메이 킹 특성을 설정합니다.

제목 일치 티켓 세션 내에서 각 구성원에 구성원 단위 특성을 지정합니다. 이러한 속성 이름을 "matchAttrs"를 사용 하 여 MultiplayerSession.SetCurrentUserMemberCustomPropertyJson 메서드를 호출 하 여 설정 됩니다. 이 호출은 특성에 배치 /members/ {인덱스} / 속성/사용자 지정/matchAttrs 티켓 세션 내에서 각 플레이어에 필드.

매치 메이 킹 프로세스 "병합" 각 당 멤버는 hopper에 대 한 UI는 Xbox Live 구성과 해당 특성에 대해 지정 된 플랫 방법에 따라 단일 티켓 수준 특성으로 합니다. 이 [XDP](https://xdp.xboxlive.com) 또는 [Windows 개발자 센터](https://developer.microsoft.com/dashboard/windows/overview)에서 구성할 수 있습니다.


### <a name="making-the-match"></a>일치를 수행합니다.

티켓 세션 및 일치를 사용 하 여 티켓 설정, 매치 메이 킹 서비스 일치 하는 다른 그룹을 나타내는 다른 티켓 세션 표현된 티켓 세션 및 만들거나 일치 대상 세션을 식별 합니다. 서비스도 일치 하는 플레이어에 대 한 예약 대상 세션에서 만들고 일치 하는 것으로 다음 티켓 세션을 표시 합니다. MPSD에는 다음 티켓 세션에 이러한 변경의 제목을 알립니다.

이제 제목, 충분 한 플레이어에 나타나는지 확인 하기 위해 대상 세션을 초기화 하 고 품질의 QoS (서비스) 검사를 서로를 성공적으로 연결할 수 있는 수행 해야 합니다. 초기화 및/또는 QoS 실패할 경우 제목 다른 그룹을 찾을 수 있도록 매치를 다시 제출 될 티켓 세션을 표시 합니다. 이 프로세스에 대 한 자세한 내용은 **대상 세션 초기화 및 QoS를**참조 하세요.

일치 활동을 동안 다음과 같은 변경 내용은 세션에 대 한 JSON 개체 적용 됩니다.

-   /servers/matchmaking/properties/system/status 필드 "찾을 수 없습니다"로 설정 합니다.
-   /servers/matchmaking/properties/system/targetSessionRef 필드 대상 세션을 설정합니다.
-   {인덱스} /members//속성/사용자 지정 멤버에 각 티켓 세션에 대 한 matchAttrs 필드 복사 / {인덱스} / / 상수/사용자 지정/matchmakingResult/playerAttrs 필드.
-   각 플레이어에 대 한 티켓 특성 /members/ {인덱스} 일치 티켓에서 ticketAttributes 필드에서 복사/상수/사용자 지정/matchmakingResult/ticketAttrs 필드.


### <a name="maintaining-the-match-ticket"></a>일치 티켓을 유지 관리

매치 메이 킹 서비스 티켓 세션의 스냅숏을 세션에 대 한 일치 티켓 생성 될 때 한 번에 사용 합니다. 따라서 모든 플레이어에 가입 하거나 티켓 세션, 제목은 삭제 하 고 다시 일치 티켓 매치 메이 킹 서비스를 사용 해야 합니다.


### <a name="filling-spots-in-an-in-progress-game-session"></a>진행 중인 게임 세션에서 채우기 지점

타이틀 자세한 이미 진행 중인 게임에 참가 하거나 라운드 완료 된 후에 게임 세션 채우기 및 일부 플레이어 왼쪽 찾기에 대 한 일치 티켓 세션으로 기존 게임 세션 다시 사용할 수 있습니다. 이 프로세스를 "백필" 이라고 합니다.

백필 수행 하는 한 가지 방법은 PreserveSessionMode.Always 설정 preserveSession 매개 변수를 사용 하 여 **MatchmakingService.CreateMatchTicketAsync 메서드** 를 호출 하 여 진행 중인 세션 "유지" 하는 일치 티켓을 만드는 것입니다. 그런 다음 매치 메이 킹 서비스는 기존 세션 매치 메이 킹 프로세스 내내 유지 되도록 하 고 결과 대상 세션 됩니다.

두 개의 세션을 항상 설정 preserveSession 결합할 수 있으므로 서로 일치 시킬 수 있을이 패턴을 단점이 있습니다. 따라서 아주 느리지만 백필 매치 메이 킹 수 있습니다. 이 시나리오를 방지 하기 위해 제목을 사용 해야 PreserveSessionMode.Always 경우 게임 상태를 유지 해야 합니다. Otherwis PreserveSessionMode.Never 설정과 일치 하는 경우는 플레이어가 새로운 targetSession로 마이그레이션.


### <a name="deleting-the-match-ticket"></a>일치 티켓 삭제

일치 티켓을 삭제 하려면 제목 **MatchmakingService.DeleteMatchTicketAsync 메서드**를 호출 합니다. 매치 메이 킹 서비스 티켓을 삭제 합니다.

1.  티켓 세션에서 사용자에 대 한 연결을 중지합니다.
2.  업데이트를 티켓 세션의 /servers/matchmaking/properties/system/status 필드 "취소".


### <a name="matchmaking-for-games-using-xbox-live-compute"></a>Xbox Live 계산을 사용 하 여 게임에 대 한 매치 메이 킹

다음 예제에서는 계산 Live 서비스를 사용 하 여 높은 수준의 매치를 보여 줍니다. 유사한 접근 방식을 타사 리소스에서 호스트 되는 게임에 적용할 수 있습니다.

1. 3.  그룹의 모든 구성원 티켓 세션에 대 한 /constants/system 개체에서 measurementServerAddresses 값을 다운로드 합니다. 각 구성원 플랫폼 API를 사용 하 여 각 주소 ping 그리고 /members/ {인덱스}에 정의 된 세션으로 기본 데이터 센터의 정렬된 된 목록 업로드/속성/시스템/serverMeasurements 합니다.

7.  매치 메이 킹 서비스 서버에 고려해 야 측정 컬렉션을 수행할 다른 그룹을 나타내는 사람과 티켓 세션을 찾습니다. 높은 기본 동일한 데이터 센터에 있는 다른 그룹을 사용 하 여 그룹 일치 하려고 합니다.

9.  모든 클라이언트는 위에서 설명한 것 처럼 대상 세션에서 초기화를 수행 합니다. 그러나 클라이언트가 계산 Live에 연결 되어, 때문에 수행 하지 않습니다 QoS 상호 연결을 확인 합니다.
10. 일부 또는 모든 클라이언트 **GameServerPlatformService.AllocateClusterAsync 메서드**를 호출 합니다. 첫 번째 다른 받게만 동안 할당을 트리거합니다. 메서드를 MPSD에서 대상 세션을 가져오고 대상 세션 뿐만 아니라 부하 및 기타 라이브 계산 관련 기술 datacenter 기본 설정에 따라 dataceter를 선택 합니다.
11. 모든 클라이언트를 재생합니다.

### <a name="configuring-team-rules"></a>팀 규칙 구성

#### <a name="using-team-rules"></a>팀 규칙을 사용 하 여

팀 규칙 구성 되 면는 hopper 내 티켓 수 없습니다에서 일치 하는 경우 분할 일으키지 않고 팀에 해당 그룹에 맞게 방법은 없습니다. 규칙은 대상 세션 멤버/상수/사용자 지정/matchmakingresult/initialTeam 아래에 결과 팀 할당을 작성 합니다. 이 동작은 사용자가 간단 하 게 제안 된 할당을-제목 참고는 플레이어를 조정 하 여 발생할 수 있습니다. 있으므로 더 나은 게임을 여전히 다른 팀으로 분할에서 티켓을 방지 하는 동안 발견할 수 있습니다.

진행 중인 게임에 대 한 티켓 만들어지면 팀 규칙 추가 정보가 필요 합니다. 예를 들어, 8 플레이어 4v4 두 경우 게임 플레이어가 놓기에 이거나 연결이 끊어진 가정 합니다. 이러한 빈 지점 입력 하려는 제목 하지만 게임을 재생 하는 동안에 팀을 나타나거나 수 없습니다.

게임 진행 중에서에 맞게 시도 "항상"으로 설정 된 PreserveSession 필드를 사용 하 여 티켓으로 표현 됩니다. 다음과 같은 경우 팀 플레이어에 게 이미 할당 하기 때문에 제목을 지정 해야 현재 팀 할당 일치는 얼마나 많은 공간을 알 수 있도록 각 팀에서 엽니다. 팀의 이름을 제공 하 각 플레이어 켜진 경우 각 플레이어 게임 세션/me/속성/시스템/그룹을 구성원에서 해당 팀 이름을 씁니다. 이 필드는 JArray입니다.

위의 속성, 게임 세션에 기록 되 면 하나의 플레이어만 티켓을 더 많은 플레이어를 찾기 위해에서에 세션을 만듭니다. 티켓 충족 되 면 일치는 다시 쓸 참여 하 고 있는 모든 플레이어의 제안 된 팀 구성원/상수/사용자 지정/matchmakingresult/initialTeam

#### <a name="managed-initialization-episodes-and-stages"></a>관리 되는 초기화 에피소드 및 단계

대상 세션 매치 메이 킹 세션에 새 플레이어를 추가 하 든 지 관리 되는 초기화를 거칩니다. 스마트 예약 됨, 즉 각 구성원 슬롯을 차지 하는 세션을 아직 가입 하지 않은 사용자 상태로 세션 멤버를 추가 합니다. 새 플레이어의 각 그룹에 새 초기화 에피소드를 트리거합니다.

초기화를 완료 되 면 각 플레이어 성공 하거나 실패 하는 프로세스입니다. 대상 세션을 사용 하 여 초기화에 성공 하면 플레이어가 재생할 수 있습니다. 실패 하면 플레이어가 다른 세션에 일치 시킬 매치 메이 킹에 다시 전달 되어야 합니다. 항상 설정 *preserveSession* 매개 변수를 사용 하 여 세션 매치를 제출한 시장의 경우 세션의 기존 멤버 거치지 않고 초기화 MPSD 가정 제대로 설정에 따라 합니다.

각 관리 되는 초기화 에피소드 이러한 단계로 구성 됩니다.

-   가입-세션 멤버 보류 사용자 상태를 활성으로 이동 하려면 세션에 스스로 작성 하 고 보안 장치 주소 등의 기본 데이터를 업로드 합니다.
-   -피어 기반 토폴로지에 대 한 세션 멤버 서로 QoS 측정 측정과 세션에 결과 업로드 합니다.
-   평가-MPSD 마지막 두 단계 결과 확인 및 세션 및/또는 멤버 성공적으로 초기화 여부를 결정 합니다.

제목 코드는 각 사용자 (및 따라서 세션)에 참여 하 고 측정 단계를 진행 하려면 세션에서 작동 합니다. 다음 수 재생을 시작 하거나로 돌아가 매치 메이 킹 후 평가 스테이지의 성공 또는 실패 했습니다.


### <a name="configuring-the-target-session-for-initialization"></a>대상 세션 초기화에 대 한 구성

제목 상수를 사용 하 여 초기화 되는 대상 세션에서 관리 되는 초기화 프로세스를 구성할 수 있습니다. 이러한 상수 버전 107, 권장된 템플릿 버전을 사용 하 여 세션 템플릿에서 /constants/system에서 설정 됩니다. 두 가지 유형의 구성 설정 만들 수 있습니다: 전체로 관리 되는 초기화 프로세스를 구성 하는 설정 및 구성 QoS 요구 사항 설정 합니다. [MPSD 세션 템플릿](multiplayer-session-directory.md) 일반적인 제목 시나리오에 대 한 세션 템플릿 예제를 참조 하세요.

| 참고                                                                                                                              |
|------------------------------------------------------------------------------------------------------------------------------------------------|
| QoS 요구 사항 대상 세션 초기화 구성에 정의 되지 않은 경우 초기화 하는 동안 측정 단계를 건너뜁니다. |


#### <a name="configuring-managed-initialization-as-a-whole"></a>전체로 초기화 관리 구성

다음은 전체 관리 되는 초기화를 제어 하기 위해 설정할 필드입니다. /Constants/system/memberInitialization 개체의 일부인:
- joinTimeout 합니다. MPSD 초기화 에피소드 시작 된 후에 가입 하려면 각 구성원에 대 한 대기 하는 기간을 지정 합니다. 기본값은 10 초입니다.
-   measurementTimeout 합니다. MPSD 측정 단계 시작 된 후 QoS 측정 결과 업로드 하려면 각 구성원에 대 한 대기 하는 시간을 지정 합니다. 기본값은 30000 초입니다.
-   membersNeededToStart 합니다. 성공 하려면 첫 번째 초기화 에피소드에 대 한 초기화 시 성공 해야 하는 구성원 수를 지정 합니다. 기본값은 1입니다.

| 참고                                              |
|----------------------------------------------------------------|
| 이 임계값이 충족 되지 않으면 모든 구성원 초기화를 실패 합니다. |


#### <a name="configuring-qos-requirements"></a>구성 QoS 요구 사항

QoS 제목 피어 투 피어 또는 피어와 호스트 간 토폴로지를 사용 하는 경우에 초기화 하는 동안 필요는 합니다. 각 토폴로지 /constants/시스템/아래 토폴로지 특정 상수에 매핑됩니다.
###### <a name="configuring-qos-requirements-for-peer-to-peer-topology"></a>피어 투 피어 토폴로지에 대 한 구성 QoS 요구 사항

| 참고                                                                                                                                                                                         |
|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 타이틀은 피어 투 피어 토폴로지에 대 한 QoS 요구 사항을 설정할 수에 대 한 경우는 드뭅니다. 이러한 설정은 매우 제한적 고 엄격한 네트워크 주소 변환 (Nat)를 사용 하 여 플레이어에 대 한 문제를 일으킬 합니다. |

피어 투 피어 토폴로지 QOS 요구 사항 peerToPeerRequirements 개체에서 설정 됩니다. 모든 클라이언트는 다른 모든 클라이언트에 연결할 수 있어야 합니다. 개체에는 다음 관련 된 필드가 있습니다.
-   latencyMaximum 합니다. 두 명의 클라이언트 간 최대 대기 시간을 지정합니다.
-   bandwidthMinimum 합니다. 두 명의 클라이언트 간의 최소 대역폭을 지정합니다.


###### <a name="configuring-qos-requirements-for-peer-to-host-topology"></a>피어-호스트 토폴로지에 대 한 구성 QoS 요구 사항

피어와 호스트 간 토폴로지 QOS 요구 사항 peerToHostRequirements 개체에서 설정 됩니다. 모든 클라이언트는 단일 일반적인 호스트에 연결할 수 있어야 합니다. 이 개체를 구성 하 고 초기화 성공 하면, MPSD "호스트 후보." 라고 알려진 잠재적 호스트는 클라이언트의 초기 목록이 만듭니다. 설정 하는 필드는 다음과 같습니다.
-   latencyMaximum 합니다. 각 피어와 호스트 간에 최대 대기 시간을 지정합니다.
-   bandwidthDownMinimum 합니다. 각 피어와 호스트 간에 다운스트림 최소 대역폭을 지정합니다.
-   bandwidthUpMinimum 합니다. 각 피어와 호스트 간에 업스트림 최소 대역폭을 지정합니다.
-   hostSelectionMetric 합니다. 호스트를 선택 하는 데 메트릭을 지정 합니다.
