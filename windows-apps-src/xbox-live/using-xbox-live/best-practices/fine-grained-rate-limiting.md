## <a name="xbox-live-fine-grained-rate-limiting"></a>Xbox Live 세분화 된 속도 제한

다른 제목 간에 공유 Xbox Live 리소스 **공정 사용량** 홍보 위임 된 **세밀 하 게 세분화 된 속도 제한** 합니다. 이 방법은 요청 엔터티는 지정 된 기간의 번호의 수를 유지 하는 서비스는 대부분의 기존 제한 시스템과 같이 있습니다.

그런 다음 서비스 지정 된 제한에 도달 하는 엔터티 있는 엔터티에서 모든 들어오는 요청 켜지 떨어져 거부 상태로 이동 합니다. 엔터티는 지정 된 기간 되돌리려면 엔터티 연결된 수를 일으키는 만료 되 면이 상태를 종료할 수만 있습니다.

하지만 세밀 하 게 세분화 된 속도 제한 사용 하 여 동일한 핵심 메커니즘 **사용자와 제목의** 조합을 추적 FGRL 엔터티를 추적 하는 대신 위에서 언급 한 및 연결된 수가으로 **두** 가지 **제한** 하 비교 하나로 미니 합니다. FGRL 이중 제한은 GameClips의 요청 수 있는지에 대 한 요청 수 영향을 주지 것입니다 즉 각 서비스에 적용 됩니다. 다음 섹션에서는 사용자 및 제목 페어링, 제한, 이중 및 HTTP 429 제한 응답 개체에 대 한 자세한 내용은 들어갑니다.

## <a name="fair-usage"></a>공정 사용

Xbox Live 같은 높은 수준의 환경에 관계 없이 합리적인이 재생 하 고 어떤 게임/앱 각 사용자가 있어야 한다고 생각 합니다. FGRL 다음 시나리오를 해결 하기 위해 설계 되었습니다.

하지만 개발자는 라이브 놓음 개발자 B가 해도 제목을 최적의 서비스 사용을 보장 하는 모범 사례이 하나에 버그를 알 수 없는 모든 Xbox를 따르는 제목을 출시 방금 했습니다. 이 버그 제목 및 각 사용자 스팸 로드가 이동 하면 서비스는 현재 상태를 발생 합니다. 서비스 느려지고 결국 문제를 발생 시킨 개발자 B 버그 된 경우에 개발자는 사용자 환경을 손상 시 키 지를 중단 합니다. FGRL 구현 된 경우 서비스 제한이 있었을 리소스 원형의 공정 슬라이스에 개발자 A의 제목을 사용할 수 있도록 제목 동작 하 고 누락에서 요청 수신을 중지할 수 있습니다.

## <a name="title-and-user-granularity"></a>제목 및 사용자 단위

제목 및 사용자는 Xbox Live 리소스의 공정 사용을 보장 하는 키로 선택 했습니다. 사용자만 추적 각 타이틀 통합의 사용자 환경을 것 시나리오를 만듭니다. 예를 들어 대부분의 제목을이 예제에서는 사용자가 서비스를 사용 하므로 이미, 5 분에 100 개 이상의 요청을 허용 하는 사용자 서비스에 설정 된 세밀 하 게 세분화 된 속도 제한 가정해 보겠습니다. 1 분에서 100 개의 요청을 수행 하는 게임을 플레이 하는 사용자가 제한을 초과 합니다 및 사용자가 사용자 서비스;에 더 많은 모든 요청을 만들 수 없습니다. 고 상상해 보세요 동일한 기간에 사용자가 다음 홈 화면으로 돌아갑니다. 그 친구 목록 클릭: 사용자가 이미 초과 했기 때문 친구 호출 나와 있는 제한 실패 하 게 5 분 간격 날짜가 될 때까지 있지만 홈 화면 respon 없습니다. 제한 된 상태에서 사용자를 추가 하기 위한 sible 합니다.

에서만 기반 제한 또는 제목은 동일 하 게 불공정 결과 생성 합니다. 제목 당 제한을 설정 제목은의 인기 무시 됩니다 있으며 요청은 방금 수 먼저 제한에 도달할 때까지 처음 제공 되었습니다.

사용자 및 제목 페어링 제목이 없는 것 보다 적절 한 리소스 원형의 일관 된 조각을 각 사용자를 제공 하는 동안 활성 사용자 수를 지정 하 여 더 많은 리소스를 사용 하도록 보장 합니다.

## <a name="burst-and-sustain-limits"></a>버스트 및 제한을 유지합니다

일반적으로 지정 된 기간 동안 추적 되는 끝점 마다 하나의 제한의 구성 속도 제한 합니다. 이 기간은 시간의 양을 나타냅니다 엔터티 요청 수를 추적 됩니다. 기간이 끝날 때 엔터티 개수 추적을 다시 시작 하는 0으로 재설정 됩니다.

하지만이 방법은 대부분의 API의 게임에 대 한 충분히 복원 된 live Xbox를 호출 하는 앱에 대해 작동 합니다. 위의 솔루션에서 일관 된 일정 예측 가능한 장 원의 사용자를 호출 하는 가정 합니다. Xbox Live 경우, 서비스 및 요청 제목/앱에 따라 호출 패턴 크게 다릅니다.

단 하나의 제한 선택이 경우 필요 호출 패턴 스펙트럼의 양쪽 끝에 손상 됩니다. Xbox Live 솔루션은 두 기간 및 제한 합니다. 더 작은 기간이 더 큰 한 Sustain 기간 이라고 하는 동안 버스트 기간 이라고 합니다. 버스트 시간 기간는 sustain은 항상 300 초 반면 FGRL는 항상 15 초 (5 분) 하므로 20 5 분 길이의 sustain 기간 동안 버스트 기간입니다. 둘 다 버스트 및 제한을 유지는 사본은 동일한 시기 및 추적 따라서 카운트 요청이 동시 합니다. 각 서비스에는 고유한 버스트 및 sustain 횟수가 의미 서비스 버스트와 sustain 제한을 설정 됩니다. 이러한 두 제한을 작동 방식을 이해를 돕기 위해 함께 아래 표에 사용자가 다양 한 FGRL 구현에 서비스의 요청 하는 타이틀을 재생 합니다. 이 경우 버스트 제한은 30 요청 15 초에서 sustain 제한 되며는 100 요청 5 분 동안 합니다.

## <a name="http-429-response-object"></a>HTTP 429 응답 개체

연결 된 사용자 및 제목 개수 하나 이상 버스트 되었거나 제한 열정 서비스 요청을 처리 하지 않습니다 및 HTTP 429 응답을 대신 반환 됩니다. "요청이 너무 많이"에 대 한 HTTP 429 코드 스탠드 "X 초 후 재시도" 값을 포함 하는 헤더와 함께 합니다. FGRL 429 호출 엔터티 다시 시도 하기 전에 대기할 시간을 지정 하는 헤더 후 다시 시도 포함 합니다. XSAPI를 사용 하는 개발자가 XSAPI를 인정 하 고 다시 시도 후 헤더 처리를 걱정할 필요가 없습니다. 실제 응답에는 다음 필드가 포함 됩니다.
