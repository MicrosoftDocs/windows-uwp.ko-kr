## <a name="xbox-live-fine-grained-rate-limiting"></a>Xbox Live 세밀 하 게 세분화 된 속도 제한

**세분화 된 속도 제한을 미세** 승격에 의뢰 했습니다 **공정한 사용** 직함 간에 공유 Xbox Live 리소스. 이 방법은 같은 엔터티의 지정 된 기간에 대 한 요청 개수를 유지 하는 서비스에 있는 대부분의 일반적인 제한 시스템입니다.

지정된 된 서비스 한도 도달 하는 엔터티는 그런 다음 엔터티에서 들어오는 모든 요청 제거 하 여 설정 됩니다는 거부 상태로 이동 됩니다. 엔터티는 지정 된 기간 다시 설정 하려면 연결 된 엔터티 수를 일으키는 만료 되 면이 상태를 종료할 수만 있습니다.

그러나 조합 추적 FGRL 하나의 엔터티를 추적 하는 대신 위에서 언급 한 동일한 핵심 메커니즘을 사용 하 여 세밀 하 게 세분화 된 속도 제한을 **사용자 및 제목** 와 연결 된 개수를 비교 **두** 다른 **제한** 하나에 따라 합니다. FGRL 이중 제한이 GameClips에 대 한 요청 수가 현재 상태에 대 한 요청 수가 영향을 주지 것입니다 즉 각 서비스에 적용 됩니다. 다음 섹션에서는 사용자 및 제목 쌍, 이중 제한 및 제한 HTTP 429 응답 개체에 대해 더 자세히 이동 됩니다.

## <a name="fair-usage"></a>공정한 사용

Xbox Live 사용자 마다 같은 높은 수준의 경험에 관계 없이 권한를 재생 하는 게임/앱을가지고 있어야 한다고 생각 합니다. FGRL는 다음 시나리오를 해결 하기 위해 설계 되었습니다.

하지만 개발자는 최상의 개발자 B 제목을 출시 했습니다도 하는 동안 서비스의 최적 사용률을 확인이 알 수 없는 버그를 라이브 모든 Xbox 뒤에 오는 제목을 출시 했습니다. 이 버그 제목 및 각 사용자 부하가 이동 하면 서비스는 현재 상태를 스팸 메일에 발생 합니다. 서비스 느려지고 결국 개발자 B의 버그 문제를 발생 시킨 것도 개발자 A의 사용자를 위한 환경을 주요를 중단 합니다. FGRL 구현 된 경우 서비스는 수 있는 리소스의 공정한 해당 조각을 개발자 A의 제목에 맞도록 작동 하므로 제목 누락에서 요청 수신을 중지 합니다.

## <a name="title-and-user-granularity"></a>제목 및 사용자 세분성

제목 및 사용자 키로 Xbox Live 리소스의 공정한 사용 되도록 선택 되었습니다. 사용자만 추적 각 타이틀 통합 오랫동안 사용자 환경을 것 시나리오를 만듭니다. 예를 들어, 대부분의 제목이 예제의 사용자 서비스를 사용 하므로 이미, 5 분 내에 100 개 요청을 허용 하는 사용자 서비스에 설정 된 세밀 하 게 세분화 된 속도 제한을 가정해 보겠습니다. 1 분에서 요청 수가 100 개를 만든 게임을 실행 하는 사용자가이 한도 초과 합니다 하 고 사용자 서비스에 더 이상 요청을 수행할 수 사용자 설정 되지 않습니다. 동일한 기간에 사용자 다음 홈 화면으로 다시 이동 하 고 클릭 자신의 친구 목록 가정해 보겠습니다. 이후 사용자는 이미 홈 화면 사용자 제한 상태로 전환 하는 일을 담당 했습니다. 경우에 간격이 지난 5 분까지 친구 목록 호출은 실패 하는 제한을 초과 했습니다.

경우에 기반 제한 또는 제목에서 동일 하 게 불공정 결과 산출 합니다. 제목의 인기도 title 별 제한을 설정 무시 되 고 요청은 방금 수 먼저 진행 먼저 제한에 도달할 때까지 제공 합니다.

사용자 및 제목의 페어링을 제목 없음에서 적합 한 항목 또한 각 사용자는 일관 된 원형 조각에 리소스를 제공 하는 동안 활성 사용자 수를 지정 하는 보다 더 많은 리소스를 사용 하 게 합니다.

## <a name="burst-and-sustain-limits"></a>초과 사용 및 한도 유지 합니다.

일반적으로 속도 제한 이루어져 지정 된 기간 동안 추적 되는 끝점 별로 하나의 제한이 있습니다. 이 기간 시간을 나타내는 엔터티 요청 수를 추적 됩니다. 기간의 끝 엔터티 수를 다시 추적을 시작 하려면 0으로 다시 설정 됩니다.

그러나이 방법은 대부분의 API의 게임에 대 한 충분 한 복원 력이 없습니다 및 Xbox를 호출 하는 앱 라이브에 대해 작동 합니다. 위의 솔루션을 일관 된 안정적인 예측 가능한 manor에서 사용자를 호출 하는 가정 합니다. 에서는 Xbox Live 서비스 및 요청 제목/앱에 따라 호출 패턴은 크게 다릅니다.

한도 선택 합니다.이 경우 해야 호출 패턴 스펙트럼의 양쪽 끝에 손상 시 키 지 합니다. ## <a name="http-429-response-object"></a>HTTP 429 응답 개체

연결 된 제목 및 사용자 수 하나 이상인 버스트 되었거나 제한 유지할 때 서비스 요청을 처리 하지 않습니다 하 고 HTTP 429 응답 대신 반환 됩니다. HTTP 429 코드 약자 "요청이 너무 많음" "X 초 후 재시도" 값을 포함 하는 헤더와 함께 합니다. FGRL 429 호출 엔터티를 다시 시도 하기 전에 대기 해야 하는 시간을 지정 하는 헤더 후 다시 시도 포함 합니다. XSAPI를 사용 하는 개발자는 XSAPI 준수 및 Retry-after 헤더를 처리 하는 대로 걱정할 필요가 없습니다. 실제 응답에는 다음 필드가 포함 됩니다.

| 필드 이름      | 값 유형 | 예                | 정의                       |
